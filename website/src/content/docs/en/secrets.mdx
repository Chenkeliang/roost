Secrets in Roost are **entirely optional**. If you only sync dotfiles, packages,
app config, and projects, there is no key to manage and none of this applies.
Secrets matter only when you choose to back up secret values (for example secret
environment variables).

## The age key is the root of trust

When you do encrypt secrets, Roost uses [age](https://github.com/FiloSottile/age)
— a small, single-file, open-source tool (no password manager required). Secret
values are encrypted to your age key, typically stored at:

```
~/.config/sops/age/keys.txt
```

<Aside type="danger" title="Back it up offline">
If you lose your age key (and any recovery copy), the encrypted data is
**unrecoverable**. Keep an offline backup.
</Aside>

## Getting the key onto a second Mac

The only real question is how a **second** Mac obtains the **same** age key.
Roost supports three backends; `bootstrap.sh` tries them in order and falls back
to generating a fresh local key if none is configured — so **none is required**:

<CardGrid>
  <Card title="(A) Manual — zero extra tools">
    Run `age-keygen` once, then copy `~/.config/sops/age/keys.txt` to the other
    Mac yourself (AirDrop / USB / Keychain).
  </Card>
  <Card title="(B) 1Password (op)">
    If you already use 1Password, store the key and set `ROOST_AGE_OP_REF`.
  </Card>
  <Card title="(C) rbw">
    The free, open-source Bitwarden CLI. Store the key and set `ROOST_AGE_RBW_REF`.
  </Card>
</CardGrid>

### Example: rbw

```sh
brew install rbw
rbw login                          # one-time: enter Bitwarden credentials
rbw add roost-age-key              # store your age private key
export ROOST_AGE_RBW_REF=roost-age-key
./scripts/bootstrap.sh             # pulls the key via rbw, then applies
```

If rbw isn't installed (or the ref isn't set), `bootstrap.sh` falls back to
generating a new age key locally.

## Three sources for a secret env var

The backends above answer "how does a second Mac get the age key". A separate,
related question is where a **secret environment variable** gets its value from
at apply time. The Aliases & Env module supports three sources per secret var
(per [ADR-0004](https://github.com/Chenkeliang/roost/blob/main/docs/adr/0004-secret-env-resolution.md)):

<CardGrid>
  <Card title="age (default)">
    You enter the value; Roost encrypts it to `roost/env-secrets/<NAME>.age` in
    the repo. On `load --apply`, the local age key decrypts it and inlines it
    into a `0600` `~/.config/roost/env.sh` — never committed in plaintext.
  </Card>
  <Card title="op:// (1Password)">
    The repo stores only an `op://Vault/Item/field` **reference** — no
    ciphertext. On apply, Roost resolves it via `op read`. Needs the `op` CLI.
  </Card>
  <Card title="rbw (Bitwarden)">
    The repo stores only the rbw entry **reference** — no ciphertext. On apply,
    Roost resolves it via `rbw get`. Needs the `rbw` CLI.
  </Card>
</CardGrid>

With `op`/`rbw` references, not even ciphertext touches the repo — only a
locator string, which is safe to show in the UI. The resolved value lives in
memory only: it's inlined into `env.sh` and discarded, never committed, logged,
or returned in an API response.

<Aside type="note" title="Failure-safe">
If `op`/`rbw` is unavailable or a lookup fails, `apply` skips that one variable
(warning by name only, never the value) instead of writing a half-finished
assignment. `roost doctor` checks `op`/`rbw` availability up front.
</Aside>

## The pre-commit secret scanner

`capture` is gated by a plaintext-secret **scanner**. It inspects non-secret
fields and function bodies, and if it detects what looks like a plaintext secret,
it **blocks** the capture and tells you to mark the offending value as `secret`
so Roost encrypts it instead — without ever logging the value.

You can also run the scan over your whole config repo at any time:

```sh
roost audit
```

It reports the count of encrypted files, scanned files, and any plaintext
findings, and exits non-zero if it finds any.

## Backing up and rotating the key

Because the age private key is the recovery material for every encrypted item,
back it up offline. `key backup` shows where it lives and how:

```sh
roost key backup            # location + offline-backup guidance
roost key backup --show     # also print the key contents, to copy somewhere safe
```

To re-encrypt every `.age` file in your repo to a new recipient (for example
after generating a fresh key):

```sh
roost key rotate --new-recipient age1...
```

`key rotate` reads the existing key from `~/.config/age/key.txt` by default; pass
`--old-key <path>` if yours lives elsewhere.

## The three nevers

Whatever the backend, Roost guarantees that secrets are:

- **never** committed to your repo in plaintext (the scanner is a hard gate),
- **never** shown in the UI, and
- **never** written to logs.
