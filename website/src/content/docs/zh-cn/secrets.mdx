Roost 中的密钥是**完全可选**的。如果你只同步 dotfiles、软件包、应用配置和项目,那就
没有密钥要管理,本页内容也都用不上。只有当你选择备份密钥值(例如密钥类环境变量)时,
密钥才有意义。

## age 密钥是信任之根

当你确实要加密密钥时,Roost 使用 [age](https://github.com/FiloSottile/age)——一个小巧、
单文件、开源的工具(无需密码管理器)。密钥值会被加密到你的 age 密钥,通常存放于:

```
~/.config/sops/age/keys.txt
```

<Aside type="danger" title="离线备份它">
若你丢失了 age 密钥(以及任何恢复副本),被加密的数据将**无法恢复**。请保留一份离线备份。
</Aside>

## 把密钥带到第二台 Mac

唯一真正的问题是:**第二台** Mac 如何获得**同一把** age 密钥。Roost 支持三种后端;
`bootstrap.sh` 会按顺序尝试它们,若都未配置,则回退到在本地生成一把全新密钥——所以
**没有哪一项是必需的**:

<CardGrid>
  <Card title="(A)手动——零额外工具">
    运行一次 `age-keygen`,然后自己把 `~/.config/sops/age/keys.txt` 拷到另一台 Mac
    (AirDrop / U 盘 / 钥匙串)。
  </Card>
  <Card title="(B)1Password(op)">
    若你已在用 1Password,把密钥存进去并设置 `ROOST_AGE_OP_REF`。
  </Card>
  <Card title="(C)rbw">
    免费、开源的 Bitwarden CLI。把密钥存进去并设置 `ROOST_AGE_RBW_REF`。
  </Card>
</CardGrid>

### 示例:rbw

```sh
brew install rbw
rbw login                          # 一次性:输入 Bitwarden 凭证
rbw add roost-age-key              # 存入你的 age 私钥
export ROOST_AGE_RBW_REF=roost-age-key
./scripts/bootstrap.sh             # 通过 rbw 拉取密钥,然后应用
```

若未安装 rbw(或未设置该引用),`bootstrap.sh` 会回退到在本地生成一把新的 age 密钥。

## 密钥类环境变量的三种来源

上面的后端回答的是"第二台 Mac 如何拿到 age 密钥"。另有一个相关但不同的问题:一个**密钥类
环境变量**在 apply 时从哪里取值。Aliases & Env 模块为每个密钥变量支持三种来源
(见 [ADR-0004](https://github.com/Chenkeliang/roost/blob/main/docs/adr/0004-secret-env-resolution.md)):

<CardGrid>
  <Card title="age(默认)">
    你输入值;Roost 把它加密为仓库里的 `roost/env-secrets/<NAME>.age`。在 `load --apply`
    时,本机 age 密钥解密它并内联进一个 `0600` 的 `~/.config/roost/env.sh`——绝不以明文入库。
  </Card>
  <Card title="op://(1Password)">
    仓库只存一个 `op://Vault/Item/field` **引用**——无密文。apply 时 Roost 经 `op read`
    取值。需要 `op` CLI。
  </Card>
  <Card title="rbw(Bitwarden)">
    仓库只存 rbw 条目**引用**——无密文。apply 时 Roost 经 `rbw get` 取值。需要 `rbw` CLI。
  </Card>
</CardGrid>

用 `op`/`rbw` 引用时,连密文都不会进仓库——只有一个定位符字符串,它在 UI 里显示是安全的。
解析出的值只存在于内存中:它被内联进 `env.sh` 后即弃,绝不入库、不进日志、不出现在 API 响应里。

<Aside type="note" title="失败安全">
若 `op`/`rbw` 不可用或某次取值失败,`apply` 会跳过那一个变量(仅按名告警,绝不显示值),
而不是写半成品赋值。`roost doctor` 会提前检查 `op`/`rbw` 的可用性。
</Aside>

## 提交前的密钥扫描器

`capture` 受一个明文密钥**扫描器**把关。它检查非密钥字段和函数体,若检测到看起来像
明文密钥的内容,就会**阻止**本次采集,并提示你把那个值标记为 `secret`,从而让 Roost
改为加密它——整个过程绝不记录该值。

你也可以随时对整个配置仓库运行扫描:

```sh
roost audit
```

它会报告已加密文件数、已扫描文件数,以及任何明文发现,并在发现任何明文时以非零退出。

## 备份与轮换密钥

由于 age 私钥是每个加密条目的恢复材料,请离线备份它。`key backup` 显示它在哪以及如何备份:

```sh
roost key backup            # 位置 + 离线备份指引
roost key backup --show     # 同时打印密钥内容,便于复制到安全之处
```

要把仓库里每个 `.age` 文件重新加密到一个新的接收方(例如在生成新密钥之后):

```sh
roost key rotate --new-recipient age1...
```

`key rotate` 默认从 `~/.config/age/key.txt` 读取现有密钥;若你的密钥在别处,请传
`--old-key <path>`。

## 三个"绝不"

无论用哪种后端,Roost 都保证密钥:

- **绝不**以明文提交到你的仓库(扫描器是硬门),
- **绝不**在 UI 中显形,
- **绝不**写入日志。
